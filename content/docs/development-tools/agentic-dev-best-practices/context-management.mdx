# Context Management & Task Decomposition

Every coding agent operates within a <Def>context window</Def> — the total amount of text it can "see" at once. Managing that window is one of the most important skills in agentic development, and it is not intuitive. The context window does not degrade like a battery slowly draining. It degrades unevenly, with accuracy dropping in the middle of long contexts and <Def>hallucination</Def>s increasing as the window fills.

---

## Context Window Sizes

| Model | Window Size | Notes |
|-------|-------------|-------|
| GPT-5.2/5.3-Codex | 400K tokens | Standard for most Codex tasks |
| GPT-5.1-Codex-Max | Multi-window compaction | Summarizes earlier context to sustain coherence across 24+ hour tasks |

400K <Def>token</Def>s is roughly 300K words or about 1,500 pages of text. That sounds like a lot — and it is. But "fits in the window" and "works well in the window" are very different things.

---

## Why Context Degrades Before the Window Fills

Two research findings are particularly relevant:

**"Lost in the Middle" (Liu et al., 2023):** When relevant information is placed in the middle of a long context, <Def>LLM</Def> accuracy drops by 30% or more compared to when the same information appears at the beginning or end. The model does not process all positions equally.

**"Context Rot" (various studies, 2024-2025):** Across 18 models tested, all showed degradation as context length increased, even when the information was not in the "lost middle" zone. Performance declines gradually, not in a cliff.

<Callout type="warning">
**Practical context degradation heuristics.** These are approximate and model-dependent, but useful as mental anchors: at around 70% of the context window, precision starts declining measurably. At 85%, hallucination rates increase noticeably. Above 90%, behavior becomes erratic — the model may ignore instructions, contradict earlier output, or generate plausible-sounding nonsense. Do not treat these as hard thresholds. Treat them as a reason to keep sessions focused.
</Callout>

## The One-Task-Per-Session Pattern

Codex Cloud's architecture aligns naturally with good context management. Each task gets:

1. A fresh sandbox with a clean clone of your repo
2. A new context window with only AGENTS.md and your task description
3. An isolated execution environment

This means every task starts with maximum context quality. The pattern is simple:

1. **Describe a discrete task** — one feature, one bug fix, one refactor
2. **Let Codex execute** in its sandbox
3. **Review the proposed <Def>PR</Def>** — check the diff, run it locally if needed
4. **Merge or request changes**
5. **Start a new session** for the next task

You never need to manually manage context with Codex Cloud because the architecture does it for you. Each session is disposable. The permanent record lives in your git history, not in the agent's memory.

<Callout type="tip">
**Git history is your handoff mechanism.** Each Codex task produces a PR with a description, a diff, and (if you wrote good task descriptions) a clear record of intent. The next session can pick up where the last left off by reading the codebase — which now includes the merged changes. You do not need to explain what happened in previous sessions.
</Callout>

## Task Decomposition Heuristics

How big should a single task be? Here are practical guidelines:

### Single Prompt (1-3 files, with tests)

Tasks that touch a small number of files and have clear boundaries work well as a single prompt.

**Example:** "Add a loading spinner to the profile page. Create a `Spinner` component at `src/components/Spinner.tsx` that accepts a `size` prop. Use it in `src/app/profile/page.tsx` while data is loading. Write tests in `src/__tests__/Spinner.test.tsx`."

This is focused: three files, clear behavior, testable output.

### Plan First, Execute in 2-3 Chunks (5+ files or new patterns)

When a feature spans multiple files or introduces a pattern the codebase has not seen before, break it into sequential tasks that each build on the last.

**Example:** "Add an authentication flow" is too broad for one task. Break it into:

1. **Auth utilities:** "Create `src/lib/auth.ts` with functions for `login`, `logout`, and `getCurrentUser`. These should call the `/api/auth/*` endpoints. Write tests."
2. **Login page:** "Create a login page at `src/app/login/page.tsx` using the auth utilities from `src/lib/auth.ts`. Include email/password form with validation. Write tests."
3. **Protected routes:** "Create a `withAuth` higher-order component at `src/components/withAuth.tsx` that redirects unauthenticated users to `/login`. Apply it to `src/app/dashboard/page.tsx`. Write tests."

Each task is self-contained, testable, and builds on the previous one.

### Avoid Over-Decomposition

Breaking work into 15 micro-prompts is counterproductive. Each task has overhead — Codex needs to read the codebase, understand the context, and produce a PR. More importantly, the agent loses the big picture when tasks are too granular. A component split across five sessions may end up inconsistent because no single session held the full design in context.

A rough guideline: aim for about 500 lines of change per task. Less than 50 lines is probably too small to justify a session. More than 1,000 lines means you should probably split.

## Four Context Failure Patterns

Simon Willison identifies four ways context goes wrong. Recognizing these helps you diagnose problems when agent output degrades:

### Context Poisoning
Incorrect information in the context causes the model to generate incorrect code. This happens when AGENTS.md contains outdated instructions, when previous conversation turns included wrong assumptions, or when the model reads misleading comments in the code.

**Example:** Your AGENTS.md says "use the `pages/` directory for routes" but you migrated to App Router six months ago. Codex creates files in the wrong directory.

### Context Distraction
Irrelevant information in the context pulls the model's attention away from the actual task. Long AGENTS.md files with sections about deployment when the task is about a UI component. Excessive code comments. Previous conversation turns about unrelated features.

**Example:** You paste 2,000 lines of database schema into the prompt for a task that only involves a frontend component. The model starts adding database calls that were never asked for.

### Context Confusion
The model conflates similar but distinct concepts from different parts of the context. Common with large codebases that have multiple patterns for the same thing, or when refactoring leaves old and new patterns coexisting.

**Example:** Your codebase has both a `UserProfile` component and a `ProfileCard` component. The agent mixes up their props and patterns, creating a hybrid that matches neither.

### Context Offloading
The model stops maintaining state it should track, relying on the (incorrect) assumption that earlier context is still fully accessible. This is the "lost in the middle" problem in practice.

**Example:** You describe a multi-step task. The agent executes the first steps correctly but "forgets" a constraint you specified at the beginning, producing code that violates a requirement stated 50,000 tokens ago.

## Session Handoff: The Two-File System

For projects where work spans multiple sessions and you need more continuity than git history alone provides, use two files:

### AGENTS.md (Permanent)
Project conventions that rarely change. Tech stack, commands, code style, file structure. Lives in version control. Updated infrequently and deliberately.

### HANDOVER.md (Living Document)
Current state of in-progress work. Updated at the end of each session. Not committed to version control (add it to `.gitignore`), or committed to a working branch only.

Here is a template:

```markdown
# HANDOVER.md

## Current State
[What was just completed. Reference the PR or commit.]

## Next Steps
1. [Next task, with enough detail to act on]
2. [Following task]
3. [Further out — less detail is fine]

## Open Questions
- [Decisions that need to be made before proceeding]
- [Things you tried that did not work and why]

## Architecture Decisions Made
- [Key choices and their rationale, so the next session does not revisit them]

## Files Recently Modified
- `src/lib/auth.ts` — auth utility functions (login, logout, getCurrentUser)
- `src/app/login/page.tsx` — login page with form validation
```

Update this at the end of each working session. The next session (whether you or the agent) reads it and has immediate context on where things stand.

## Good vs Bad Task Descriptions

The quality of your task description is the single biggest factor in output quality. Here are concrete examples:

### Bad: Too Vague

> "Make the app better."

Codex has no idea what "better" means. It might refactor random files, add unnecessary features, or change styling. There is no way to evaluate whether it succeeded.

### Bad: Too Broad

> "Add authentication, user profiles, settings page, and admin dashboard."

This is four features in one prompt. Even if Codex attempts all of them, it will run out of context quality before finishing. The PR will be massive and difficult to review. If one part is wrong, you cannot merge the parts that are right.

### Good: Specific and Bounded

> "Add a login form component at `src/components/LoginForm.tsx` that accepts email and password fields and calls the `/api/auth/login` endpoint on submit. Include client-side validation (email format, password minimum 8 characters) and display server error messages below the form. Write tests in `src/__tests__/LoginForm.test.tsx` covering: successful submission, validation errors, and server error display."

This names the file, describes the behavior, specifies validation rules, defines the API endpoint, and lists what the tests should cover. Codex can execute this without guessing, and you can evaluate the output against clear criteria.
