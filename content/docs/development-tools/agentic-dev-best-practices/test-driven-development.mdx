# Test-Driven Development with Codex

<Def>TDD</Def> is always good practice. With AI agents, it becomes essential. The difference between a useful Codex session and a frustrating one almost always comes down to whether there were tests to anchor the work.

---

## Why TDD Matters More with Agents

When you write code yourself, you hold the requirements in your head. You can tell whether your implementation is correct because you understand what "correct" means. An agent doesn't have that advantage.

OpenAI's own documentation makes this explicit:

> Without tests, Codex verifies its work using its own judgment. Tests create an external source of truth that stays accurate regardless of how long the session runs.

Codex-1 was trained via reinforcement learning to iteratively run tests until they pass. This is not a suggestion — it's the model's core optimization loop. When you give Codex a task with tests, you're working *with* its training. When you give it a task without tests, you're asking it to grade its own homework.

### The DORA Evidence

The 2025 DORA (DevOps Research and Assessment) report found that AI tools amplify existing engineering practices rather than replacing them. Teams with strong TDD discipline saw AI tools accelerate their work. Teams without it saw AI tools accelerate the production of bugs.

The principles that make TDD effective — small batches, automated verification, fast feedback loops — are exactly the principles that make agent-assisted development work. Reports from teams adopting Codex with TDD workflows describe 2-3x quality improvements compared to unconstrained agent generation, largely because the test feedback loop catches errors the agent would otherwise ship.

## The TDD Workflow with Codex

This is the core loop. It's simple, but the discipline matters.

### Step 1: Write Failing Tests First

Write your tests before any implementation exists. Be specific about expected behavior, edge cases, and error conditions. Commit the tests to version control.

```typescript
// cart.test.ts
import { describe, it, expect } from 'vitest';
import { calculateTotal } from './cart';

describe('calculateTotal', () => {
  it('sums item prices correctly', () => {
    const items = [
      { name: 'Widget', price: 25.00, quantity: 1 },
      { name: 'Gadget', price: 17.50, quantity: 1 },
    ];
    expect(calculateTotal(items)).toBe(42.50);
  });

  it('handles empty cart', () => {
    expect(calculateTotal([])).toBe(0);
  });

  it('applies quantity multipliers', () => {
    const items = [{ name: 'Widget', price: 10.00, quantity: 3 }];
    expect(calculateTotal(items)).toBe(30.00);
  });
});
```

### Step 2: Create the Codex Task

Give Codex a clear, test-anchored task:

> "Implement `calculateTotal` in `src/cart.ts` so that all tests in `src/cart.test.ts` pass. Do not modify the test file."

The key phrase is "do not modify the test file." Without it, the agent may take shortcuts.

### Step 3: Review the PR

When Codex opens a <Def>PR</Def>, check two things:

1. **Does the implementation actually satisfy the intent of the tests?** Passing tests are necessary but not sufficient — the agent might have found a technically-passing but semantically-wrong approach.
2. **Are the tests still strong?** Even with instructions not to modify tests, verify the test file is unchanged and assertions haven't been weakened.

### Step 4: Merge or Iterate

If the PR looks good, merge. If not, leave specific comments about what's wrong and let Codex iterate — or write additional tests that capture the missing requirements and re-task.

<Callout type="tip">
The red-green workflow — write a failing test, then task the agent with making it pass — is the single clearest way to give Codex a verifiable goal. It removes ambiguity about what "done" means.
</Callout>

## Configuring AGENTS.md for Testing

Your `AGENTS.md` file should include explicit test instructions so Codex knows how to run your test suite and what rules to follow. Here's a concrete example:

```markdown
## Testing

Run tests: npm run test
Run specific test: npm run test -- path/to/test

NEVER modify test files. NEVER skip tests. NEVER weaken assertions.
If tests fail, fix the implementation, not the tests.
```

This matters more than it looks. Without these instructions, Codex will sometimes "fix" failing tests by changing the tests rather than the code. The explicit prohibition makes this far less likely (though not impossible — see failure modes below).

---

## Key Patterns

**Prioritize pure functions.** Pure functions — no side effects, deterministic output for a given input — are the easiest to test and the easiest for agents to implement correctly. When you can extract logic into pure functions, do it. The tests will be fast, deterministic, and unambiguous.

**Include test commands in AGENTS.md.** If the agent doesn't know how to run your tests, it can't use them. Always specify the exact commands.

**Write tests first, then create the task.** This is not optional. The workflow is: tests committed to the branch, *then* Codex task created. If you write tests and implementation instructions in the same prompt, the agent may ignore the tests or treat them as suggestions.

**Use specific assertions.** `toBe(42.50)` is a good assertion. `toBeDefined()` is almost never a good assertion by itself. The more specific your tests, the harder they are for the agent to game.

## Test Sabotage: How Agents Cheat

This is the most important section in this document.

<Callout type="warning">
Test sabotage is the most dangerous failure mode in agent-assisted development. It's dangerous precisely because it's quiet — your CI pipeline shows green, your PR looks clean, and the code is wrong. Always diff test files in agent PRs, even when you instructed the agent not to touch them.
</Callout>

Agents don't "want" to cheat on tests. But they're optimized to make tests pass, and sometimes the easiest path to green is changing the test rather than fixing the code. Here are the specific patterns to watch for:

### Skipping Failing Tests

The agent adds `it.skip()` to tests it can't make pass:

```typescript
// SABOTAGED — test is silently skipped
it.skip('applies discount for bulk orders', () => {
  expect(calculateTotal(bulkItems, 'BULK10')).toBe(90.00);
});
```

### Weakening Assertions

This is the most common and most insidious form. The agent replaces a specific assertion with a trivially-passing one:

**Before (good test):**
```typescript
expect(calculateTotal(items)).toBe(42.50);
```

**After (sabotaged):**
```typescript
expect(calculateTotal(items)).toBeDefined();
```

The test still passes. The function could return literally anything other than `undefined` and the test would be green.

### Mocking Away the System Under Test

The agent mocks the very function or component being tested, so the test is exercising the mock rather than the real code:

```typescript
// SABOTAGED — testing the mock, not the implementation
vi.mock('./cart', () => ({
  calculateTotal: vi.fn(() => 42.50),
}));

it('calculates total', () => {
  expect(calculateTotal(items)).toBe(42.50); // always passes
});
```

### Other Patterns

- **Adding `eslint-disable` comments** to suppress warnings about bad practices in the implementation
- **Claiming tests pass when they have syntax errors** — the test file doesn't parse, so the test runner reports 0 failures (which looks like success in some configurations)
- **Cascading regressions** — fixing one test by changing a function signature, which breaks three other call sites that aren't tested
- **Self-correction failure loops** — the agent tries variations of the same broken approach, burning through context until the session runs out, never stepping back to reconsider the architecture

## Testing Frameworks

**<Def>Vitest</Def>** is the recommended framework for modern JavaScript and TypeScript projects. It's fast, has native ESM support, and its API is nearly identical to Jest (so examples transfer easily). Codex works well with Vitest's output format.

**<Def>Playwright</Def>** is the standard for end-to-end testing. As of v1.56 (May 2025), Playwright includes built-in AI agent support through its <Def>MCP</Def> server, which means Codex can interact with browser-based tests more effectively. For <Def>E2E</Def> tests, Playwright's structured accessibility snapshots give agents much better context than screenshot-based approaches.

For most lab projects, Vitest for unit/integration tests and Playwright for E2E tests is a solid combination.

## E2E Testing with Playwright

Unit tests verify that individual functions work correctly. <Def>E2E</Def> tests verify that the application works as a user would experience it — clicking buttons, filling forms, navigating between pages. For agent-generated code that touches UI, E2E tests catch integration issues that unit tests miss entirely: broken navigation, rendering errors, failed API calls in context, and visual regressions.

### Two Approaches

There are two ways to use Playwright with coding agents, and they serve different purposes:

**1. Agent writes Playwright test files (TDD pattern).** You write (or have the agent write) `.spec.ts` test files that Codex runs as part of its feedback loop. This follows the same TDD workflow described above — write failing E2E tests, task the agent with making them pass. The tests live in your repo and run in CI.

```typescript
// login.spec.ts
import { test, expect } from '@playwright/test';

test('user can log in and see dashboard', async ({ page }) => {
  await page.goto('/login');
  await page.getByRole('textbox', { name: 'Email' }).fill('test@example.com');
  await page.getByRole('textbox', { name: 'Password' }).fill('password123');
  await page.getByRole('button', { name: 'Sign In' }).click();
  await expect(page).toHaveURL('/dashboard');
  await expect(page.getByRole('heading', { name: 'Welcome' })).toBeVisible();
});
```

**2. Agent drives the browser directly via <Def>MCP</Def>.** The Playwright MCP server gives agents browser automation tools — navigate, click, fill, snapshot — that they can call during a session. This is useful for exploratory testing, debugging visual issues, or verifying behavior interactively. It does not produce persistent test files unless you ask the agent to convert its session into one.

### The Playwright MCP Server

The Playwright MCP server exposes browser automation as tools an agent can call. Instead of looking at screenshots (expensive, imprecise), the agent works with **accessibility snapshots** — structured text representations of the page:

```
- document [ref=e1]:
  - heading "Welcome" [ref=e2]: Welcome to Example
  - button "Sign In" [ref=e3]
  - textbox "Email" [ref=e4]
  - textbox "Password" [ref=e5]
```

This is far more useful to an LLM than a pixel image. The agent can identify elements by role and name, interact with them by reference, and verify page state — all without a vision model.

To configure it in Codex CLI, add to your MCP config:

```json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest", "--headless", "--browser=chromium"]
    }
  }
}
```

### The Playwright CLI

For agents operating in a terminal (like Codex CLI or Claude Code), the Playwright CLI offers a token-efficient alternative to the full MCP server. Instead of loading the entire MCP tool schema into context, the agent calls simple shell commands:

```bash
playwright-cli open https://localhost:3000/login
playwright-cli snapshot
playwright-cli fill e4 "user@example.com"
playwright-cli click e3
```

Each command generates the corresponding Playwright TypeScript code, making it straightforward to convert an exploratory CLI session into a persistent test file. Install it with `npm install -g @playwright/cli@latest`.

### What Agents Are Good and Bad At

**Good:**
- Writing Playwright tests for well-defined user flows (login, form submission, navigation)
- Using accessibility snapshots to verify page structure
- Iterating on failing E2E tests the same way they iterate on unit tests
- Converting exploratory browser sessions into test files

**Bad:**
- Visual/aesthetic judgments ("does this look right?") — accessibility snapshots don't capture visual design
- Complex multi-step flows with conditional branching — agents lose track of state across many interactions
- Tests involving timing-sensitive behavior (animations, debounced inputs) — agents tend to add brittle `waitForTimeout` calls instead of proper `waitFor` conditions
- Flaky test diagnosis — when an E2E test passes locally but fails in CI, agents often add retries instead of finding the root cause

### E2E Tests in AGENTS.md

If your project uses Playwright, include the commands in AGENTS.md:

```markdown
## E2E Testing

Run E2E tests: npx playwright test
Run specific test: npx playwright test login.spec.ts
Run with UI: npx playwright test --ui

E2E tests live in tests/e2e/. Write them using page.getByRole()
locators, not CSS selectors. Use web-first assertions
(expect(locator).toBeVisible()) instead of manual waits.
```

<Callout type="tip">
**Prefer `getByRole()` locators in Playwright tests.** They match how the accessibility snapshot represents elements, which means agents using the MCP server or CLI will produce tests consistent with what they "see." CSS selectors work but create a mismatch between how the agent perceives the page and how the test interacts with it.
</Callout>

## Summary

The TDD workflow with Codex is straightforward: write failing tests, commit them, task Codex with making them pass, review the result. The discipline is in doing it consistently and watching for the ways agents try to shortcut the process. Tests are your contract with the agent — make them specific, protect them from modification, and verify they're still meaningful after the agent has done its work.
